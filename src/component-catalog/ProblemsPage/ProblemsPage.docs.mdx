# ProblemsPage

`ProblemsPage` is a full-page organism component that renders a searchable,
filterable, and paginated browser for coding problems. It is built entirely with
**Material UI** and follows the project's Component-Driven Development (CDD)
conventions.

---

## Architecture

### Hook — `useProblemsPage`

All filter state and derived data live in the `useProblemsPage` hook so the
component template stays declarative and easy to test in isolation.

```ts
const {
  filteredProblems, // Problem[] — result of applying all active filters
  solvedCount, // number — problems with status === 'solved'
  totalCount, // number — full dataset length
  tagOptions, // string[] — unique tags derived from all problems
  filterState, // FilterState — { searchQuery, difficultyFilter, statusFilter, tagFilter }
  handleSearchChange,
  handleDifficultyChange,
  handleStatusChange,
  handleTagChange,
  handleClearFilters,
  drawerOpen,
  handleDrawerOpen,
  handleDrawerClose,
} = useProblemsPage();
```

**Search debounce** — the raw `searchQuery` is debounced by **300 ms** inside
`useProblemsPage` before it is applied to `filteredProblems`, preventing
expensive re-renders on every keystroke.

**Tag options** — derived at hook initialisation via:

```ts
const tagOptions = useMemo(() => Array.from(new Set(PROBLEMS.flatMap((p) => p.tags))).sort(), []);
```

## Filtering Logic

All four filters are applied with a logical `&&` chain. A problem must satisfy
every active criterion to appear in the list.

```ts
PROBLEMS.filter(
  (p) =>
    matchesSearch(p, debouncedSearch) &&
    matchesDifficulty(p, difficultyFilter) &&
    matchesStatus(p, statusFilter) &&
    matchesTag(p, tagFilter)
);
```

| Filter     | Default | Options                             |
| ---------- | ------- | ----------------------------------- |
| Search     | `""`    | Free-text substring (title)         |
| Difficulty | `"All"` | All · Easy · Medium · Hard          |
| Status     | `"All"` | All · Solved · Attempted · Unsolved |
| Tag        | `"All"` | All · _derived from dataset_        |

| ---------- | --------------- | -------- |

### Status Icons

| Status      | Icon component             | Colour     |
| ----------- | -------------------------- | ---------- |
| `solved`    | `CheckCircleOutlineIcon`   | `success`  |
| `attempted` | `RemoveCircleOutlineIcon`  | `warning`  |

Desktop table rows and mobile list items where `status === 'solved'` receive a
**4 px left border** in `theme.palette.success.main` and a subtle
`theme.palette.action.selected` background tint — matching the Figma reference.

## Responsive Behaviour

| Breakpoint                | Filter UI               | Problem UI                |
| ------------------------- | ----------------------- | ------------------------- |
| `xs` / `sm` (< 900 px)    | Filters button → Drawer | `List` + `ListItemButton` |
| `md` and above (≥ 900 px) | Inline filter bar       | `Table` with 6 columns    |

`minHeight: theme.spacing(5.5)` on every interactive element).

## Styling Conventions

- **No hardcoded `px` values** — all spacing uses `theme.spacing()`.
- **No external CSS files** — all styles live in `ProblemsPage.styles.ts` via
  MUI `styled()` components or exported `sx` objects.
- `styled()` components accept a `isSolved` transient prop
  (`shouldForwardProp`) to avoid forwarding it to the DOM.
